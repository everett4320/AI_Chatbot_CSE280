## Commands

```bash
npm install          # install dependencies
npm run dev          # dev server with HMR at http://localhost:6173 (NOT 5173 ‚Äî see vite.config.ts)
npm run build        # production build into ./build (client + server bundles)
npm run start        # serve ./build/server/index.js via @react-router/serve
npm run typecheck    # runs `react-router typegen` then `tsc` ‚Äî run after editing routes
```

No test runner, linter, or formatter is configured. TypeScript strict mode is on; `npm run typecheck` is the only static check. To verify a change end-to-end: `npm run typecheck && npm run build`.

Docker: `docker build -t ai-chatbot-lehigh . && docker run -p 3000:3000 ai-chatbot-lehigh` (container `CMD` is `npm run start`).

## Architecture

This is a **React Router v7 framework-mode** app (`ssr: true` in `react-router.config.ts`) ‚Äî React Router here is the full stack: routing, loaders/actions, SSR, and Vite build integration all come from `@react-router/dev`. Do not confuse with React Router as a client-only library.

**Route surface is tiny on purpose.** `app/routes.ts` declares a single `index("routes/home.tsx")`. The homepage is a static landing page that mounts `<ChatWidget />`. All real UX lives inside the floating chat widget ‚Äî adding features typically means extending the chat tree, not adding routes.

**Chat state flow.** `app/hooks/use-chat.ts` is the single source of truth ‚Äî `messages`, `isLoading`, `error`. It uses `loadingRef` + `messagesRef` so `sendMessage` has a stable identity but sees current state. There is **no persistence** (refresh wipes the conversation) and no streaming ‚Äî responses arrive as one string. Message shape is `{ id, role: "user" | "assistant", content, timestamp }` (`app/types/chat.ts`).

**Component tree (all in `app/components/`):** `ChatWidget` owns open/close state and the `useChat()` instance ‚Üí `ChatArea` is the 400√ó620 panel (header, scrollable log, error banner, input) ‚Üí `ChatMessage` renders assistant content via `react-markdown` and exports `ASSISTANT_BUBBLE` so `TypingIndicator` can match the bubble style ‚Üí `ChatInput` is an auto-growing textarea (capped at 140px) with Enter-to-send, Shift+Enter for newline.

**API boundary ‚Äî important.** `app/services/chat-api.ts` is the only network code. It `POST`s `{ messages: [{role, content}, ‚Ä¶] }` to `import.meta.env.VITE_CHAT_API_URL` and expects `{ reply: string }` back. **If `VITE_CHAT_API_URL` is unset, it returns a hardcoded stub after 600ms** ‚Äî that's what you see in local dev. Create a gitignored `.env` with `VITE_CHAT_API_URL=‚Ä¶` and restart `npm run dev` to hit a real backend.

Note: this payload shape does **not** match the upstream AWS endpoint documented in `../fetched_site/` (`action: "question"` against `https://8lyrpsdez5.execute-api.us-east-1.amazonaws.com/call`). Wiring up the real backend requires rewriting the request/response mapping in `chat-api.ts`; the stub shape was invented here and is not a contract with any deployed service.

**Styling = Tailwind v4 with a custom theme in CSS.** `app/app.css` declares tokens via `@theme { ‚Ä¶ }` ‚Äî use `lehigh-navy`, `lehigh-navy-dark`, `lehigh-mint`, `lehigh-surface`, `lehigh-brown`, `lehigh-brown-dark` as Tailwind color utilities. There is **no** `tailwind.config.*` file; theme lives entirely in CSS. Inter is preloaded in `app/root.tsx`.

**Path alias.** `~/*` ‚Üí `./app/*` (declared in both `tsconfig.json` and `vite-tsconfig-paths`). Prefer `~/components/foo` over relative imports.

**Generated types.** `.react-router/types/` is autogenerated by `react-router typegen` and feeds `./+types/*` imports (e.g. `import type { Route } from "./+types/home"`). It's gitignored ‚Äî run `npm run typecheck` after adding/changing routes so these regenerate. Never hand-edit files under `.react-router/`.

## Repo context

This directory is the **custom frontend** portion of a larger capstone repo. The parent `../` holds:
- `knowledge_base/` ‚Äî RAG content (markdown)
- `fetched_site/` ‚Äî reference implementation captured from `dev-le-chat.cc.lehigh.edu`, including the real API payload format and a batch-test question suite
- `scripts/` ‚Äî asset fetch + test suite runners (`run_question_suite.sh`)

When the intended API contract is unclear, `../fetched_site/README.md` and `../fetched_site/dist/bundle-pretty.js` are the authoritative reference for the upstream backend.
